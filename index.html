Change Log


Welcome

The Creative Little UI Kit is a performance focused, ITCSS based super scaleable front-end framework. It's built for the modern web and only supports modern browsers, that's right we've left IE10- out in the cold, all for the benefit of web users everywhere. It includes a flex based fluid grid, common, comprehendible Javascript plugins and a tasty minimalist style for all your friendly neighbourhood web elements.


Installation

Installing the UI Kit is very simple, just download the latest release from Github and start coding. There is no package manager to setup, it doesn't rely on node.js it will work out of the box as any good CSS framework should.

The UI Kit was built for the team at Creative Little Dots to use in-house and as such is biased to a specific setup. Although it's not a requirement we use the fantastic CodeKit app in-house to do all our compiling, linting, minification, transpiling and image optimisation.


Compatibility

The UI Kit really is built for the modern web and as such only supports the cool kid browsers... and that kid who sits in the corner and eats flies.

ITCSS

What?

Inverted Triangle CSS, created by Harry Roberts, is a new architecture that uses the core principles and design of the CSS language to help utilise the code we write in a way that benefits everyone, whether you're the first developer or the 6th developer to work on a project.

The philosophy behind ITCSS is to order CSS itself by measurable properties defined in the language, such as specificity. This results in an almost purely class based methodology throwing shade to ID's and occasionally giving them a good kicking when no one is looking.

Using ITCSS is fairly simple, once you learn the basic rules behind the philosophy you can carry on as usual with your code dilly dally.


Why?

We are now at a point in web development where even sometimes simple projects are complicated and the biggest plague to hit front-end development since it's inception is code rot. Adding styles to the end of your style sheet because you can't be bothered to work out it's rightful place has become the nightmare of any developer working on a project second hand.

By using the ITCSS architecture we have a component based framework that attempts to prevent code rot and keep CSS maintainable.


How?

For those of you who detest reading, and I've got to say i'm amazed you've made it all the way to paragraph 10! We would recommend hiring Harry Roberts himself to teach you his Jedi ways.

For the rest of you here's a quick rundown of how to get started using ITCSS in your next project. The system works by having a main SCSS stylesheet purely for including partials. It is ordered based on specificity and begins with files that don't output and code, these would be your mixins, functions and variables.

Our first level after settings and tools is our generic level. This level is used to declare things such as fonts, keyframes and most importantly shared styles using psuedos and wildcard selectors. (*, ::selection, etc.)

Following those we create any element level styles using tag selectors (input, h1, img, etc.) these are low specificity so they come before our classes. Any code in this level should be considered global or default for the tag selectors.

Next up we have the object level, this is similar to components but with one key difference, objects have no aesthetics, they are used to create layouts and bind elements to the page, the grid for example is an object.

Before we get to components, we add in our third party CSS from any plugins we need so we can override them in components without being too specific. It's worth noting that in some cases it may be better to switch components and parties if the third party CSS is already very specific (Uses selectors like this: #myElem > * > p > a.hello).

Components! Finally, this where you'll spend most of your time. Keep your components dry and don't attach them to locations. For example creating a component that can only live on a specific page such as "contact" is not reusable!

Of course as much as we try to prevent it theres always going to be cases where we need !important or can only override by matching the selector tree. For this we use trumps. Trumps can also be used for helpers and atomic style classes.


Settings

The UI Kit is configurable by overriding the default settings. In the "styles > framework > settings" folder there are a handful of files. You can use this as a direct reference to find out what settings are available to you.

global.scss
$radius:            3px         !default;
$btnRadius:         0           !default;
$btnReset:          true        !default;
$speed:             0.25s       !default;
